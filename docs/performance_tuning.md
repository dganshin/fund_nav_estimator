# 性能优化与架构决策 (Performance & Architecture)

本文档记录了系统中关于性能、缓存与并发的核心设计决策，方便后续维护者理解“为什么这么做”以及避免踩坑。

---

## 1. 实时估值高频刷新与缓存控制 (LIVE_BUNDLE_TTL)

### 背景
用户希望在盘中以较高的频率（如 1秒 或 2秒）获取最新的基金估值，并实时反映在盘中折线图上。

### 优化前的问题
`LIVE_BUNDLE_TTL` 曾被硬编码为 `12.0` 秒。前端如果设置为 1秒 刷新：
- 请求到达后端后，后端在 12 秒内都会命中 `LIVE_BUNDLE_CACHE` 的内存缓存。
- 导致前端折线图每隔 12~14 秒才会出现一次数据跳动，无法实现真正的秒级实时更新。

### 改进方案
将 `LIVE_BUNDLE_TTL` 下调为 `0.8` 秒。
- **为何是 0.8?**：确保在前端 1秒 的轮询频率下，每次请求到达后端时缓存刚好失效，从而触发底层数据源（如腾讯、AKShare 行情接口）的真实请求。
- **数据源保护**：为防止过于高频的请求导致 IP 被封禁，在前端 UI 增加了 `2秒` 的刷新档位，作为性能和实时性的最佳平衡点。

## 2. 盘中数据 (Intraday Snapshots) 的异步持久化

### 背景
系统需要记录每天 9:30-11:30 和 13:00-15:00 期间每只基金的分钟级估值涨跌百分比，供前端绘制盘中分时图。为了防止服务重启丢失当天数据，这些打点数据必须写入 SQLite 数据库的 `intraday_snapshots` 表。

### 优化前的问题
如果在提供 `/api/live-estimates` 接口响应的主线程中同步执行 SQLite 的 `INSERT / UPDATE`：
- 数据库 I/O 延迟会直接叠加在 API 响应时间上。
- 在并发查询或磁盘繁忙时，会导致前端页面刷新出现明显的卡顿。

### 改进方案：无锁异步写入
在 `src/frontend_app.py` 的 `_record_intraday` 模块中采用了**异步线程剥离写入**的策略：

1. **内存秒级响应**：估值结果直接写入内存变量 `INTRADAY_SERIES`，并使用 `INTRADAY_LOCK` 轻量级锁保证线程安全。API 读取分时图时，直接从内存变量返回，速度极快（微秒级）。
2. **后台异步落库**：当检测到数据更新（例如新的一分钟，或者同一分钟内数值发生了变化），把这些 SQL 更新操作打包，新开一个独立的 `threading.Thread(target=_write_db, daemon=True)` 在后台去慢慢写 SQLite。
3. **数据压缩**：数据库中只存储 `estimate_pct`（百分比），同一分钟内的多次请求只触发 `UPDATE`，不会无限制 `INSERT`。前端折线图里的金额是通过百分比 × 持仓金额动态算出的。

**收益**：彻底隔离了 DB I/O 对主流程 API 的影响，即使 1 秒高频刷新，也依然能保持毫秒级响应不卡顿。

---

## 3. 服务器重启时的数据重建

为了配合异步落库，系统在启动（初始化）时，会自动执行 `_load_intraday_from_db()`：
- 自动清理 `trade_date < today` 的历史过期盘中数据，防止 `intraday_snapshots` 表无限膨胀。
- 将当天的已有打点数据完整加载回内存的 `INTRADAY_SERIES`。
这样服务随时重启，盘中折线图都能无缝衔接。
